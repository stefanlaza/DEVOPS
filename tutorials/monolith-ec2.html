Java Monolith
to EC2 / Tomcat

Jenkins builds a Spring Boot WAR with Maven, uploads the artifact to S3, and Ansible performs a rolling deploy across four EC2 Tomcat instances behind an ALB — zero downtime, automatic health check verification.

Spring Boot Jenkins Maven S3 Ansible EC2 Tomcat ALB
Overview

Deployment Architecture

This pattern deploys a traditional Java WAR application to EC2 Tomcat servers behind an ALB. The rolling deploy ensures at least 3 of 4 servers are always serving traffic during deployment. Ansible handles the orchestration — draining each server from the load balancer before updating it.

Jenkins Build

Maven compiles, runs tests, packages WAR file.

S3 Artifact

Versioned WAR uploaded to S3. Previous versions retained for rollback.

Ansible Deploy

Rolling update — one server at a time. Drain, update, health check, re-register.

ALB

Health checks confirm each server is healthy before next server is updated.

Why Tomcat over Docker

This pattern is common in enterprises that haven't containerized yet. Tomcat on EC2 is simpler to operate, easier to debug, and familiar to teams coming from Java EE backgrounds. The Ansible rolling deploy gives zero-downtime deployments without container orchestration complexity.

Build Guide

01

Spring Boot WAR Packaging

Configure Spring Boot to package as a WAR instead of the default executable JAR. This requires extending SpringBootServletInitializer and marking the embedded Tomcat as provided.

<!-- pom.xml — key changes for WAR packaging -->
<packaging>war</packaging>

<dependencies>
  <!-- Mark embedded Tomcat as provided — external Tomcat will be used -->
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-tomcat</artifactId>
    <scope>provided</scope>
  </dependency>
</dependencies>
// ServletInitializer.java — required for external Tomcat deployment
@SpringBootApplication
public class Application extends SpringBootServletInitializer {

    @Override
    protected SpringApplicationBuilder configure(SpringApplicationBuilder builder) {
        return builder.sources(Application.class);
    }

    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
02

Jenkins Pipeline — Build and Upload to S3

Jenkins builds the WAR and uploads it to S3 with the build number in the key. Every build is preserved — rollback means deploying any previous build number.

pipeline {
  agent any
  environment {
    S3_BUCKET = 'company-artifacts'
    AWS_REGION = 'us-east-1'
    APP_NAME = 'monolith-app'
  }

  stages {
    stage('Build') {
      steps {
        sh 'mvn clean package -P production -B -q'
      }
      post {
        always { junit 'target/surefire-reports/*.xml' }
      }
    }

    stage('Upload Artifact to S3') {
      steps {
        sh """
          WAR_FILE=target/${APP_NAME}-*.war

          # Upload with build number in path for versioning
          aws s3 cp \$WAR_FILE \
            s3://${S3_BUCKET}/releases/${APP_NAME}/${BUILD_NUMBER}/app.war \
            --region ${AWS_REGION}

          # Also tag as 'latest' for convenience
          aws s3 cp \$WAR_FILE \
            s3://${S3_BUCKET}/releases/${APP_NAME}/latest/app.war \
            --region ${AWS_REGION}

          echo "Artifact uploaded: build ${BUILD_NUMBER}"
        """
      }
    }

    stage('Deploy via Ansible') {
      steps {
        withCredentials([sshUserPrivateKey(credentialsId: 'ec2-deploy-key', keyFileVariable: 'SSH_KEY')]) {
          sh """
            ansible-playbook ansible/rolling-deploy.yml \
              --inventory ansible/inventory/production \
              --private-key \$SSH_KEY \
              --extra-vars "s3_bucket=${S3_BUCKET} build_number=${BUILD_NUMBER} app_name=${APP_NAME}" \
              --diff
          """
        }
      }
    }

    stage('Smoke Test via ALB') {
      steps {
        sh """
          ALB_DNS=\$(aws elbv2 describe-load-balancers \
            --names ${APP_NAME}-alb \
            --query 'LoadBalancers[0].DNSName' \
            --output text)

          # Hit health endpoint 3 times to confirm stable
          for i in 1 2 3; do
            curl -sf --max-time 10 http://\$ALB_DNS/app/actuator/health
            sleep 2
          done
          echo "Smoke tests passed"
        """
      }
    }
  }

  post {
    failure {
      echo "Build ${BUILD_NUMBER} failed. Previous version still running."
    }
  }
}
03

Ansible Rolling Deploy Playbook

The playbook deploys to one server at a time. For each server: deregister from ALB, wait for connections to drain, download WAR from S3, deploy, start Tomcat, wait for health check to pass, re-register with ALB. Only then move to the next server.

---
# ansible/rolling-deploy.yml
- name: Rolling deploy to Tomcat
  hosts: tomcat_servers
  serial: 1              # One server at a time
  max_fail_percentage: 0 # Stop immediately if any server fails

  vars:
    tomcat_home: /opt/tomcat
    app_context: app
    aws_region: us-east-1

  pre_tasks:
    - name: Deregister server from ALB target group
      delegate_to: localhost
      become: false
      shell: |
        aws elbv2 deregister-targets \
          --target-group-arn "{{ target_group_arn }}" \
          --targets Id="{{ ansible_host }},Port=8080" \
          --region {{ aws_region }}

    - name: Wait for active connections to drain (30 seconds)
      pause:
        seconds: 30

  tasks:
    - name: Stop Tomcat
      systemd:
        name: tomcat
        state: stopped

    - name: Remove old WAR and exploded directory
      file:
        path: "{{ item }}"
        state: absent
      loop:
        - "{{ tomcat_home }}/webapps/{{ app_context }}.war"
        - "{{ tomcat_home }}/webapps/{{ app_context }}"

    - name: Download new WAR from S3
      command: >
        aws s3 cp
        s3://{{ s3_bucket }}/releases/{{ app_name }}/{{ build_number }}/app.war
        /tmp/{{ app_name }}-{{ build_number }}.war
        --region {{ aws_region }}

    - name: Deploy WAR to Tomcat webapps
      copy:
        src: "/tmp/{{ app_name }}-{{ build_number }}.war"
        dest: "{{ tomcat_home }}/webapps/{{ app_context }}.war"
        owner: tomcat
        group: tomcat
        mode: '0644'
        remote_src: yes

    - name: Start Tomcat
      systemd:
        name: tomcat
        state: started

    - name: Wait for application to pass health check
      uri:
        url: "http://localhost:8080/{{ app_context }}/actuator/health"
        status_code: 200
        return_content: yes
      register: health_result
      retries: 30
      delay: 10
      until: health_result.status == 200 and '"status":"UP"' in health_result.content

  post_tasks:
    - name: Re-register server with ALB
      delegate_to: localhost
      become: false
      shell: |
        aws elbv2 register-targets \
          --target-group-arn "{{ target_group_arn }}" \
          --targets Id="{{ ansible_host }},Port=8080" \
          --region {{ aws_region }}

    - name: Wait for ALB to confirm server healthy
      delegate_to: localhost
      become: false
      shell: |
        for i in $(seq 1 20); do
          STATUS=$(aws elbv2 describe-target-health \
            --target-group-arn "{{ target_group_arn }}" \
            --targets Id="{{ ansible_host }},Port=8080" \
            --query 'TargetHealthDescriptions[0].TargetHealth.State' \
            --output text --region {{ aws_region }})
          echo "Target health: $STATUS (attempt $i)"
          [ "$STATUS" = "healthy" ] && exit 0
          sleep 15
        done
        echo "ERROR: Target never became healthy"
        exit 1

    - name: Log successful deployment
      debug:
        msg: "Server {{ ansible_host }} successfully updated to build {{ build_number }}"
Important

max_fail_percentage: 0 means if any server fails its health check, Ansible stops immediately. The remaining servers are left on the old version. Fix the issue, then re-run the playbook — it will only update the servers that still have the old version.

04

EC2 Tomcat Setup (User Data)

The EC2 launch template installs and configures Tomcat via user data. This ensures every server is identical and a replacement server is ready automatically.

#!/bin/bash
set -euo pipefail

# Install JDK 17
yum install -y java-17-amazon-corretto-headless

# Create tomcat user
useradd -r -s /sbin/nologin tomcat

# Download and install Tomcat 10
cd /opt
wget -q https://dlcdn.apache.org/tomcat/tomcat-10/v10.1.18/bin/apache-tomcat-10.1.18.tar.gz
tar xzf apache-tomcat-10.1.18.tar.gz
mv apache-tomcat-10.1.18 tomcat
chown -R tomcat:tomcat /opt/tomcat
rm apache-tomcat-10.1.18.tar.gz

# Systemd service for automatic restart
cat > /etc/systemd/system/tomcat.service <<'UNIT'
[Unit]
Description=Apache Tomcat 10
After=network.target

[Service]
Type=forking
User=tomcat
Group=tomcat
Environment=JAVA_HOME=/usr/lib/jvm/java-17-amazon-corretto
Environment=CATALINA_HOME=/opt/tomcat
Environment="JAVA_OPTS=-Xms512m -Xmx1536m -XX:+UseG1GC -XX:MaxGCPauseMillis=200"
ExecStart=/opt/tomcat/bin/startup.sh
ExecStop=/opt/tomcat/bin/shutdown.sh
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
UNIT

systemctl daemon-reload
systemctl enable tomcat
systemctl start tomcat

# Install SSM agent for Systems Manager access (no SSH keys needed)
yum install -y amazon-ssm-agent
systemctl enable amazon-ssm-agent
systemctl start amazon-ssm-agent
05

Rollback Procedure

Rollback is the same deploy command with a previous build number. Every build is stored in S3 — rollback to any previous version in under 10 minutes.

# Rollback to build 142 (last known good)
ansible-playbook ansible/rolling-deploy.yml \
  --inventory ansible/inventory/production \
  --private-key ~/.ssh/ec2-deploy.pem \
  --extra-vars "s3_bucket=company-artifacts build_number=142 app_name=monolith-app"

# List available builds in S3
aws s3 ls s3://company-artifacts/releases/monolith-app/ --recursive | grep app.war | sort

# Verify current version running on all servers
ansible tomcat_servers -i ansible/inventory/production \
  -m uri \
  -a "url=http://localhost:8080/app/actuator/info return_content=yes" \
  --become false | grep version
Rollback Speed

Because the WAR is already in S3 and servers have Tomcat installed, a rollback is just a re-deploy of the old artifact. Total rollback time: approximately 8 minutes for 4 servers with 30 second drain per server.
